# Contributing to the PaintPower Engine

Thank you for your interest in contributing to the **PaintPower Engine**!  
This project is still in early development, so every contribution helps shape its future.

This document explains how to set up the project, how to contribute, and what to expect when submitting pull requests.

### Remember that the PaintPower Engine uses C# 10.0!

---

## 🛠️ Getting Started

### 1. Fork the repository
Click **Fork** on GitHub to create your own copy of the project.

### 2. Clone your fork
```
git clone https://github.com/<your-username>/PaintPower-Engine.git
```

### 3. Install requirements
PaintPower is written in **C#**.

You can use either:

#### **Visual Studio 2022**
- Open the solution file  
- Select the *PaintPower* project  
- Build and run

#### **Visual Studio Code**
- Install the **C# Dev Kit** extension  
- Open the folder  
- VS Code will load the project automatically  
- Press Run

---

## 🧪 Running the Project

Once opened in your IDE:
- Build the project  
- Run the *PaintPower* executable  
- If something fails, check the Issues page or open a new issue

## Coding Guidelines

- Use clear, descriptive names  
- Keep code readable and commented  
- Follow (most of the) C# conventions (PascalCase for classes, camelCase for variables)  
- Keep PRs focused on a single change or feature  

- And absolutely no sound editors!

---

## Reporting Bugs

If you find a bug:
1. Check if it already exists in Issues  
2. If not, open a new issue  
3. Include:
   - Steps to reproduce  
   - Expected behavior  
   - Actual behavior  
   - Screenshots if helpful  

Use the label **bug**.

---

## Suggesting Features

Feature ideas are welcome!  
Open an issue with the label **enhancement** and describe:

- What the feature does  
- Why it’s useful  
- How it might be implemented  

---

## Submitting Pull Requests

1. Create a new branch:
```
git checkout -b feature/my-feature
```

2. Make your changes  
3. Commit with a clear message  
4. Push your branch  
5. Open a Pull Request to the **main** branch  
6. Wait for review  

PRs should:
- Be focused  
- Pass build checks  
- Follow coding guidelines  

---

## Issue Labels

- `good first issue` — beginner-friendly tasks  
- `help wanted` — tasks open to contributors  
- `bug` — something broken  
- `enhancement` — improvements or new features  
- `documentation` — docs-only changes
